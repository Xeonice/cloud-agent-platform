## ADDED Requirements

### Requirement: Multi-route research fan-out before proposal
The propose flow SHALL run a parallel research fan-out before generating `proposal.md`, covering at least three independent routes: web research (external practices and libraries), codebase scan (existing architecture, integration points, reusable patterns), and prior-change archive scan (`openspec/changes/archive`). The fan-out SHALL synthesize results into a `research-brief.md` written into the change directory.

#### Scenario: Research brief is produced before proposal
- **WHEN** the deep-propose workflow runs for a change
- **THEN** a `research-brief.md` exists in the change directory before `proposal.md` is generated
- **AND** the brief contains findings attributed to each of the three routes (web, codebase, archive)

#### Scenario: Research routes run in parallel
- **WHEN** the research fan-out executes
- **THEN** the three route agents are dispatched concurrently within a single phase (not serially)

### Requirement: Testable scenarios in generated specs
The deep-propose flow SHALL generate spec scenarios in Given-When-Then form that are concrete enough to be independently verifiable, so that the downstream verify phase has a checkable contract.

#### Scenario: Generated scenarios are verifiable
- **WHEN** specs are generated by the deep-propose flow
- **THEN** every requirement has at least one `#### Scenario` with explicit WHEN/THEN conditions
- **AND** no scenario relies on non-observable terms (e.g. "fast", "clean") without a measurable criterion

### Requirement: Track-annotated draft tasks
The deep-propose flow SHALL produce `tasks.md` with tasks grouped under explicit `Track` headers, each annotated with a `depends` list, as a best-effort draft for the apply phase to correct.

#### Scenario: Tasks carry track metadata
- **WHEN** `tasks.md` is generated
- **THEN** every task belongs to a named `## Track: <name>` section
- **AND** each track declares `depends:` (possibly `none`) identifying prerequisite tracks

### Requirement: Backbone is not modified
The deep-propose flow SHALL NOT modify the `openspec` CLI, the `spec-driven` schema, or the artifact dependency graph. It MUST operate by reading and writing artifacts only.

#### Scenario: No schema or CLI mutation
- **WHEN** the deep-propose workflow completes
- **THEN** no files under the openspec CLI installation or schema definition are changed
- **AND** all outputs are artifact files within the change directory
