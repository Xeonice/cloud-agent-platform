## Context

Today the per-task AIO sandbox container is created with `HostConfig.AutoRemove: true` (`aio-sandbox.provider.ts:179`) and torn down with `stop({t:0})` + `remove({force:true})` (`:231-244`). The moment a task reaches a terminal state, the Docker daemon auto-removes its `cap-aio-<taskId>` container — and the codex `rollout-*.jsonl` session record written inside it (NOT on a mounted volume) is destroyed with it. An operator who opens a terminal task's session page sees nothing reconstructable: there is no read-only replay of what the agent did, said, or ran. `forceFail` (`guardrails.service.ts:544-572`) is the single chokepoint for all 5 abnormal terminal causes (deadline / idle / circuit-breaker / abnormal-exit / provision-failed); `onTerminal` (`:442-460`) handles the clean completed path. Both call `teardownSandbox`.

Two side-car artifacts ground this design and are explicitly NOT tracked OpenSpec artifacts: `research-brief.md` (three research routes — Web prior art, Codebase code points, Archive idioms) and `spike-findings.md` (live spikes against the local compose backend with a real ChatGPT codex login; every claim observed, not inferred). The rebuilt `design-baseline/history-replay-preview.html` is the visual contract for the replay UI (5 lifecycle states, 5 filter presets, final-answer / commentary / tool-call treatments).

Live spikes confirmed the load-bearing facts: codex 0.131 (connect-in TUI mode, NOT `codex exec`) persists one `rollout-*.jsonl` per session under `~/.codex/sessions/YYYY/MM/DD/`, flushed per turn, so every turn completed before an abnormal interruption is already on disk; flipping `AutoRemove:false` + stop-only teardown leaves the stopped container's filesystem frozen 100% (rollout + workspace + `~/.codex`) even after SIGKILL, because killing processes does not destroy the filesystem; and `docker cp` (dockerode `getArchive`) reads the frozen layer out of the STOPPED container with no restart. A single one-turn container's writable layer was ~106 MB, of which `~/.codex` cache + `logs_*.sqlite` was 92 MB and the rollout only 52 KB — so trimming the cache before stop drops the kept footprint to ~15 MB.

This change retains the container for a bounded window and exposes a structured, read-only session-history replay so a completed/cancelled/failed task can be reviewed after the fact, without resuming and without touching the live terminal pipeline.

## Goals / Non-Goals

**Goals:**
- Retain a terminal task's `cap-aio-<taskId>` container in a STOPPED state (not removed) so its frozen `rollout-*.jsonl` survives for later read-only replay, at BOTH terminal chokepoints (`onTerminal` + `forceFail`), including SIGKILL'd abnormal exits.
- Expose a NEW read-only `GET /tasks/:id/session-history` endpoint that docker-cp's the rollout out of the stopped container, parses RolloutItems into a phase-keyed structured render contract, and returns a discriminated, honest 5-state response.
- Render the replay on the terminal-state branch of `$taskId.tsx` as a read-only structured transcript (rollout primary, `session.log` cold-replay secondary), matching the design baseline.
- Bound the retained footprint (pre-stop cache trim + `auth.json` clear) and bound disk growth (multi-policy retention cleaner: age + free-disk high-water-mark).
- Fix the bootstrap reap so a Dokploy redeploy / api restart cannot wipe retained history.

**Non-Goals:**
- **Resume-run is verified-but-deferred, NOT shipped** (decision A). Bare `docker start` fails (AIO `run.sh` has a non-idempotent one-shot init); the working `docker commit` + `--entrypoint /opt/gem/entrypoint.sh` path is proven in the spike but is a follow-up. This change ships read-only first.
- NO change to the live WebSocket / PTY / write-lease pipeline. The docker-cp read endpoint is a SEPARATE REST surface; the terminal-state branch is the only frontend seam that gains the replay.
- NOT modifying `audit-history` or `resource-metrics` — the session-history read is a new endpoint and contract, not a change to the audit log or per-task resource samples.
- NO multi-replica / distributed-lock support for the retention cleaner (single-instance assumption, stated explicitly).
- NO migration of existing already-removed containers — there is nothing to recover; retention applies prospectively.

## Decisions

### D1 — Retention model: flip `AutoRemove` + split teardown to stop-only (RESOLVED)
Create containers with `HostConfig.AutoRemove: false` and split `teardownSandbox` so the terminal path STOPS but does NOT `remove`. Applied at both chokepoints. The slot is still freed (slot release is independent of container removal) — only the `remove` call is suppressed. This is **BREAKING** for any consumer assuming a terminal task's container no longer exists; mitigated by the corrected reap (D5) which keeps the container set bounded.
- Alternative considered — keep `AutoRemove:true` and copy the rollout to a mounted volume during the run via a codex Stop lifecycle hook (Langfuse's `codex-observability-plugin` model). **Rejected:** the Stop hook fires only in flows that run hooks and is unreliable on SIGKILL — the exact abnormal-exit case this change targets — whereas the frozen stopped layer survives SIGKILL. Also must NEVER pass `--ephemeral` (it would suppress rollout persistence entirely). docker-cp from the frozen stopped layer is the robust choice.

### D2 — Primary source = structured rollout via docker-cp; secondary = `session.log` cold-replay (RESOLVED, decision D)
The `GET /tasks/:id/session-history` endpoint reads `rollout-*.jsonl` out of the stopped container via dockerode `getContainer(id).getArchive()` (untar the tar stream, glob `rollout-*.jsonl`, NOT `history.jsonl`). This is net-new but on the already-established dockerode client (provider + resource-sampler already use `new Docker()`, `listContainers`, `getContainer`, `.remove/.start/.stop`); `getArchive` is documented-reliable against a stopped container precisely BECAUSE `AutoRemove:false`. The SECONDARY source — `session.log` cold-replay into a read-only xterm — already exists on the durable orchestrator volume (`SnapshotManager.buildReconnectFrames`, independent of the container) and needs no docker-cp.
- Why rollout is primary, not `session.log`: for a clean exit `session.log` cold-replays into a full colored scrollback, but for an abnormal stop it is only a half-painted TUI frame — which is exactly why the structured rollout is the faithful context source. `session.log` is kept as a secondary tab for the colored-terminal view on clean exits and the "已停止(终端中断画面)" interrupted-frame state on cancels.

### D3 — Categorization keys off the `phase` field, not message ordering (RESOLVED, load-bearing)
The commentary-vs-final-answer split keys off the explicit `phase` field on assistant `output_text` blocks: `phase == 'final_answer'` → green-tinted "最终回答"; any other phase → muted-italic commentary. This is NOT a "last assistant message" heuristic — anchored on `masonc15/codex-transcript-viewer`'s `parser.py` as cited prior art. `function_call` → tool-call card (name/arguments/`call_id`); `function_call_output` → tool-output LINKED by matching `call_id`; `event_msg token_count` → inline token count on the tool-call card; user prompts may carry a developer/instruction wrapper that MUST be split off (on a known delimiter) before display. The 5 filter presets (默认/无工具/用户/答案/全部, where 答案 = user||final) and CSS treatments are fixed by the design baseline.
- Build, not buy: no off-the-shelf component (Langfuse exports data but has no embeddable viewer) gives an in-console read-only transcript for this multi-tenant operator UI, so building the structured renderer per the baseline is justified; prior art (8+ OSS rollout viewers) is the rendering contract, not invented categories.

### D4 — Honest 5-state degradation, no fabrication (RESOLVED)
Map contract terminal statuses (`TERMINAL_TASK_STATUSES` = completed/failed/cancelled/agent_failed_to_start; `provision_failed` is a `forceFail` CAUSE that lands the task in `failed`) to the baseline's 5 states: `completed`→rollout; `cancelled`→rollout + interrupted-terminal frame; `failed`→rollout-to-failure; `agent_failed_to_start` and `provision_failed`-with-no-rollout→EMPTY with reason; expired/reaped (container removed by the cleaner)→EMPTY aged-out. Not-running / expired / no-rollout are explicit STATES, never errors — mirroring the `GET /tasks/:taskId/metrics` "not-running-is-a-state" contract. The endpoint NEVER fabricates transcript content for an empty state. A `SessionHistoryResponse` discriminated schema is added to `@cap/contracts` and validated on the client with Zod `.parse`.

### D5 — Reap only RUNNING orphans (RESOLVED, the redeploy footgun)
The `onApplicationBootstrap` reap currently force-removes ALL `cap-aio-*` containers ("every such container is an orphan"). Routed as a MODIFIED delta to the existing guardrails "Startup recovery reclaims orphaned tasks and re-offers queued tasks" requirement: the reap now removes ONLY RUNNING orphans, sparing STOPPED/retained history containers by filtering on container STATE (only RUNNING) + the `cap-aio-*` identity + age, following the standard Kubernetes-GC / Docker-prune pattern (running resources with active owners are PROTECTED; orphan/terminated-phase resources are GC-eligible; age-gate prevents premature deletion). Without this, a Dokploy redeploy or api restart wipes the kept history.

### D6 — Multi-policy retention cleaner, the FIRST retention enforcer (RESOLVED)
There is NO existing retention sweeper today (the retention-days setting is stored but unenforced), so this cleaner is net-new. Modeled on `CodexDeviceLoginService`'s unref'd `setInterval` + `docker getContainer().remove({force})`. It removes a stopped `cap-aio-*` container when ANY policy trips: Policy 1 (age) — stopped age > configured retention window, read from account settings (`DEFAULT_RETENTION_DAYS = 30`, resolved `stored?.retention ?? 30`); Policy 2 (free-disk high-water-mark) — when host free disk drops below a floor, evict OLDEST-stopped first until free disk recovers, even for containers younger than the window (age alone cannot bound disk under a burst: ~15 MB × daily volume could still blow a 160 GB VPS). It NEVER removes a RUNNING container, carries an in-process `isRunning` overlap guard, and assumes single-instance (no distributed lock — a multi-replica deployment would require one). Injects `SettingsService`/`PrismaService` to read retention, mirroring how `GuardrailsService` already injects prisma for the slot ceiling.

### D7 — Pre-stop cache trim + `auth.json` clear (RESOLVED, decisions B + F-security)
BEFORE the stop — while the container is still running and `/v1/shell/exec` is reachable — trim `/home/gem/.codex` (delete the codex cache + `logs_*.sqlite`, KEEP `sessions/` + the workspace), dropping ~106 MB → ~15 MB. As cheap defense-in-depth ALSO clear/zero `/home/gem/.codex/auth.json` so the retained stopped container holds no usable credential. The path is `/home/gem/.codex` (gem user, uid 1000, HOME=/home/gem) — NOT `/root/.codex`, which is never read. The trim runs over the same `/v1/shell/exec` channel the provider already uses, but only BEFORE stop (after stop the exec surface is gone), so it slots into `teardownSandbox` before the stop call. A trim/clear failure SHALL NOT block the stop+retain — the retain is the load-bearing outcome. Trade-off (accepted): a future resumed task re-fetches the model cache on its first turn. The session-history endpoint additionally MUST NOT export `auth.json` even though the container is the trust boundary.

### D8 — New endpoint follows the `GET /tasks/:taskId/metrics` convention (RESOLVED)
`GET /tasks/:id/session-history` follows `MetricsController`'s controller shape: covered by the global `APP_GUARD` (no auth exemption), returns a discriminated response, and treats not-running/expired/no-rollout as honest STATES not errors. Frontend data plumbing follows the strict real/mock capability seam exactly like `taskResourceQuery`: a new `queryKeys.sessionHistory(id)` tuple, a `sessionHistoryQuery` with `queryFn = isCapable(domain) ? real.getSessionHistory(id) : mock(...)`, `real.getSessionHistory` using `request(path)` + the contracts `.parse`, and a capability flag. The replay component plugs into the existing terminal-state seam in `$taskId.tsx` (the page already branches on `task.status` and `canStop` is already false for terminal tasks, so "no operation controls" is already satisfied).

## Risks / Trade-offs

- **[Retained containers consume disk on a 160 GB VPS]** → Bounded by the pre-stop trim (~15 MB/container vs ~106 MB), the 30-day age policy, AND the free-disk high-water-mark policy that evicts oldest-stopped first under a burst. The cleaner is the FIRST enforcer, so the disk model rests on it working — verified by the overlap-guarded single-instance sweep.
- **[BREAKING: consumers assuming a terminal task's container is gone]** → Audited the only consumers (the bootstrap reap and the cleaner) and made them explicitly state-aware; the reap now spares stopped containers and the cleaner is the intended remover. Documented as BREAKING in the spec.
- **[`getArchive` is net-new and untar'd in-process]** → It is on the already-established dockerode client and reads a frozen layer (no restart, no live-path coupling); failures degrade to the honest expired/no-rollout empty state, never a 500.
- **[A redeploy could wipe history]** → D5 reaps only RUNNING orphans; a stopped retained container survives a process restart (explicit scenario).
- **[Multi-replica Dokploy could double-reap / race the sweep]** → Single-instance assumption stated explicitly in the spec; in-process `isRunning` guard prevents overlap within one instance. A multi-replica deployment would require a distributed lock (called out as an Open Question, not solved here).
- **[Pre-stop trim could delete the rollout by mistake]** → The trim KEEPS `sessions/` (which holds `rollout-*.jsonl`) by construction; a trim failure does not block stop+retain, so a flaky exec surface degrades to a larger footprint, never a lost rollout.
- **[Credential left in a retained container]** → `auth.json` cleared before stop AND never exported by the endpoint; the injected ChatGPT token also auto-refreshes, so a zeroed `auth.json` is inert.
- **[Apply-time descope scar — declaring "read-only" then finding a WS delta mid-apply, as cockpit-redesign did]** → Pre-committed up front: read-only-only, resume deferred, docker-cp endpoint a separate REST surface that NEVER touches the live WS/PTY/lease path; the only frontend seam is the already-existing terminal-state branch.

## Migration Plan

- **Deploy order:** ship the backend (provider + guardrails + endpoint + contract) and frontend together; the contract is additive (new `SessionHistoryResponse` schema, new endpoint) and the frontend is gated behind a capability flag, so a frontend-without-backend renders the honest empty/degraded state rather than crashing.
- **No data migration:** retention applies prospectively. Containers already auto-removed before this ships are unrecoverable (nothing to migrate); newly-terminal tasks start retaining immediately on the new code.
- **First-redeploy behavior:** the corrected reap (D5) takes effect on the first bootstrap after deploy — RUNNING orphans from the pre-deploy process are reclaimed, stopped retained containers (none yet on the first deploy) are spared thereafter.
- **Rollback:** revert restores `AutoRemove:true` + stop+remove teardown and the blanket reap. Any containers retained while the new code was live become orphans the reverted blanket reap removes on next bootstrap — acceptable, since rollback explicitly abandons retention. The new endpoint/contract/flag are inert dead code under rollback (no consumer once the flag is off).
- **Operational guardrail:** confirm the cleaner's free-disk floor against the target VPS free space before enabling, so the high-water-mark policy has headroom above the floor.

## Deferred scope

- **`session.log` cold-replay secondary source (终端回放 tab).** D2 framed `session.log`
  cold-replay into a read-only xterm as the SECONDARY replay source. The operator
  explicitly deferred the session-log work ("会话日志…等我后面有需求了再做…先聚焦回看")
  to keep this change focused on the structured conversation transcript. So the
  终端回放 tab ships as a PLACEHOLDER ("终端回放待接入") — the two-tab structure of the
  approved design is preserved, but the cold-replay implementation (a provider
  `readSessionLogFromContainer` + a read-only xterm cold-replay) is a deferred
  follow-up. The spec requirement + scenario were amended to match this agreed
  scope; the primary structured-transcript replay is the full in-scope deliverable.

## Open Questions

- Control-frame bridge for task completion (verify spec-defect, routed here NOT as a code task): the MODIFIED `frontend-console` requirement carries the inherited scenario "Control frame bridges back into the query cache", which asserts a control frame *indicating task completion* arrives over the WebSocket and is bridged into the TanStack Query cache via `queryClient.setQueryData(['tasks', id], …)`/invalidation. Re-traced against the actual realtime path: `ControlFrameSchema` (`packages/contracts/src/control-frame.ts:29-50`) has NO `task.completed` (or any completion) frame type, `session-terminal.tsx`'s `handleControl` switch (`:241-305`) has no completion case and no cache bridge, and there is NO `setQueryData` anywhere in `apps/web/src`. Completion instead propagates via the 5s-polled `taskQuery` (`queries.ts:84`, `refetchInterval: 5000`) plus a socket-`open` `invalidateQueries(queryKeys.task(id))` (`$taskId.tsx:147-156`) — a divergence from the spec's wording, not a wired-up `task.completed` frame. This is a SPEC-DEFECT, not a code task for session-sandbox-retention: this change is categorically scoped to NOT touch the live WebSocket/PTY/write-lease pipeline (Goals/Non-Goals; D8; the "descope scar" risk), and the scenario describes pre-existing live-path behavior owned by the realtime-terminal change. The scenario as written is untestable against this change's deliberately-untouched live path. Resolution belongs to the realtime-terminal spec: either add a real completion control frame and bridge it, or amend the scenario to match the polling+socket-lifecycle propagation that actually ships. Out of scope to fix here.
- Multi-replica deployment: if Dokploy ever runs >1 api replica, the cleaner needs a distributed lock (DB `findOneAndUpdate`-style) and the reap needs to coordinate so two instances don't both reap on bootstrap. Out of scope here (single-instance assumption stated in the spec); revisit if/when horizontal scaling is introduced.
- Free-disk floor + cleaner period values: the exact floor (minimum free GB) and sweep interval are tuning parameters; sensible defaults ship, but they may need adjustment against real burst volume on the production VPS.
- Resume-run follow-up: the verified `docker commit` + `--entrypoint /opt/gem/entrypoint.sh` resume path (re-inject fresh codex auth, re-attach PTY, take a slot) is deferred to a separate change; this design only guarantees the retained container is resume-capable, not that resume is wired.
