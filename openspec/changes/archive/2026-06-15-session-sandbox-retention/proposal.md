## Why

Today the per-task AIO sandbox container is created with `HostConfig.AutoRemove: true` and torn down with `stop + remove`, so the moment a task reaches a terminal state its container — and the codex `rollout-*.jsonl` session record inside it — is destroyed. Operators land on a terminal task's session page and see nothing reconstructable: there is no read-only replay of what the agent actually did, said, or ran. We want to retain the stopped container for a bounded window and expose a structured, read-only session-history replay (per the rebuilt `design-baseline/history-replay-preview.html`), so a completed, cancelled, or failed task can be reviewed after the fact without resuming or touching the live terminal pipeline.

> Side-cars: `research-brief.md` and `spike-findings.md` are referenced throughout and are explicitly NOT tracked OpenSpec artifacts.

## What Changes

- **Retain the container instead of removing it.** Flip `HostConfig.AutoRemove` from `true` to `false` and split `teardownSandbox` into a **stop-only** path (no `remove`) for retention, applied at both terminal chokepoints (`guardrails` `onTerminal` natural completion and `forceFail` for all 5 abnormal causes). **BREAKING** for any consumer that assumes a terminal task's container no longer exists.
- **Add a read-only structured session-history replay.** New `GET /tasks/:id/session-history` endpoint reads the frozen `rollout-*.jsonl` out of the STOPPED container via dockerode `getContainer(id).getArchive()` (docker-cp from the frozen layer; reliable because `AutoRemove:false`), parses RolloutItems into the design-baseline's render contract, and returns a discriminated response. Categorization keys off the assistant `output_text` `phase == 'final_answer'` field (NOT message ordering); `tool_call`↔`tool_output` link by `call_id`; user prompts split off any developer/instruction wrapper before display.
- **Render the replay on the terminal-state branch of the session page.** The `$taskId.tsx` terminal-state seam gains a read-only structured transcript (sticky filter presets 默认/无工具/用户/答案/全部; final-answer green-tint "最终回答"; commentary muted italic; tool-call bordered card with inline token count) as the PRIMARY source, with `session.log` cold-replay (already on a durable volume) as the SECONDARY terminal-replay tab. No operation controls (already satisfied: `canStop` is false for terminal tasks).
- **Honest 5-state degradation, no fabrication.** Map contract terminal statuses to the design-baseline states: `completed`→rollout; `cancelled`→rollout + interrupted-terminal frame; `failed`→rollout-to-failure; `agent_failed_to_start`→empty; expired/reaped→empty; `provision_failed` (a `forceFail` cause, lands the task in `failed`) surfaces as failed-with-no-rollout. Not-running / expired / no-rollout are explicit states, never errors.
- **Add a retention cleaner (the FIRST retention enforcer — net-new).** An unref'd periodic sweep (modeled on `CodexDeviceLoginService`) removes stopped `cap-aio-*` containers past the retention window, reading retention days from account settings (default 30). Multi-policy: also evict oldest stopped containers first when free disk drops below a high-water-mark floor (age alone can blow the disk under burst). Carries an in-process `isRunning` overlap guard; single-instance assumption stated explicitly.
- **Fix the bootstrap reap so a redeploy cannot wipe history.** The `onApplicationBootstrap` reap currently force-removes ALL `cap-aio-*` containers ("every such container is an orphan"). Change it to remove only RUNNING orphans, sparing stopped/retained history containers via state + `cap-aio-*` label + age filtering, so a Dokploy redeploy / api restart preserves retained sessions.
- **Pre-stop cache trim + security hardening (cheap defense-in-depth).** Before stop, trim `/home/gem/.codex` (drop caches + `logs_*.sqlite`, keep `sessions/`) via the existing `/v1/shell/exec` path while the container is still running, and clear/zero `/home/gem/.codex/auth.json`. The session-history endpoint MUST NOT export `auth.json`.

Decision idioms (per `spike-findings.md`, lettered options with `CHOICE: RESOLVED →`): **A** read-only-only — resume-run is verified-but-deferred, NOT shipped; **B** clean-cache trim trade-off accepted; **D** dual-source with rollout primary / `session.log` secondary (the Stop-lifecycle-hook capture alternative is considered-and-rejected: unreliable on SIGKILL — the exact abnormal-exit case we target — and `--ephemeral` must never be passed); **F-security** clear `auth.json` before stop. The docker-cp read endpoint stays a separate REST surface, never touching the live WS/PTY/lease path, to avoid the cockpit-redesign apply-time descope scar.

## Capabilities

### New Capabilities
- `session-sandbox-retention`: Per-task sandbox containers are retained (stop-only, no remove) after a task reaches a terminal state for a settings-driven window; a multi-policy retention cleaner (age + free-disk high-water-mark, overlap-guarded, single-instance) reaps stopped `cap-aio-*` containers, and a pre-stop trim of `/home/gem/.codex` (drop caches/`logs_*.sqlite`, keep `sessions/`, clear `auth.json`) bounds retained footprint.
- `session-history-replay`: A read-only `GET /tasks/:id/session-history` endpoint reads `rollout-*.jsonl` from the stopped container via dockerode `getArchive`, parses RolloutItems into a structured render contract (phase-keyed final-answer vs commentary, `call_id`-linked tool_call/tool_output, wrapper-stripped user prompts), and returns a discriminated, honest 5-state response; the console renders it as a read-only structured transcript on the terminal-state branch with `session.log` cold-replay as the secondary source.

### Modified Capabilities
- `aio-sandbox-execution`: "Per-task AIO Sandbox container provisioning" requirement changes from `AutoRemove` enabled + `teardown = stop + remove` to `AutoRemove: false` + stop-only teardown with a pre-stop `/home/gem/.codex` cache trim and `auth.json` clear.
- `guardrails`: "Wall-clock deadline force-fails a task", "Idle ceiling reclaims wedged tasks", and "A terminal sandbox exit transitions the task and frees its slot" requirements change their `teardownSandbox()` calls to stop-only (container retained, slot still freed); "Startup recovery reclaims orphaned tasks and re-offers queued tasks" is MODIFIED so the bootstrap reap removes only RUNNING orphans (sparing retained stopped containers). A new retention-cleaner requirement is added here.
- `frontend-console`: "Session page renders the live terminal and controls" and "Session page design-revision layout" requirements gain a read-only structured session-history replay mode on the terminal-state branch (honest 5-state degradation, rollout-primary / `session.log`-secondary), with a new `@cap/contracts` `SessionHistoryResponse` schema and the real/mock capability seam plumbed.

> Note: `audit-history` and `resource-metrics` are deliberately NOT listed as modified. The session-history read is a new endpoint and contract, not a change to the audit log or per-task resource samples; their requirements are unaffected.

## Impact

- **Backend hot files (single-owner under parallel-apply, per `tasks.md` partition rationale):**
  - `apps/api/src/sandbox/aio-sandbox.provider.ts` — `AutoRemove:true`→`false` (`:179`); `teardownSandbox` split to stop-only (`:231-244`); pre-stop cache trim + `auth.json` clear on `/home/gem/.codex` (`:426`) via `/v1/shell/exec`; `onApplicationBootstrap` reap → RUNNING-orphans-only with label + age filter (`:261-287`); first use of dockerode `getArchive` against a stopped container.
  - `apps/api/src/guardrails/guardrails.service.ts` — `onTerminal` (`:442-460`) and `forceFail` (`:544-572`) teardown become stop-only; new retention cleaner wired here, injecting `SettingsService`/`PrismaService` (default 30d via `settings-logic.ts:43/81`).
- **New backend surface:** `GET /tasks/:id/session-history` controller following the `GET /tasks/:taskId/metrics` convention (`metrics.controller.ts:33-36`) — global `APP_GUARD` auth, discriminated response, not-running-is-a-state-not-an-error.
- **Contracts:** new `SessionHistoryResponse` schema in `packages/contracts/src` (RolloutItem → UI item mapping, 5-state discriminator).
- **Frontend:** `apps/web/src/routes/_app/tasks/$taskId.tsx` terminal-state branch (`:115-121`, `:178-196`) gains the replay component; new `queryKeys.sessionHistory(id)` + `sessionHistoryQuery` (`queries.ts`), `real.getSessionHistory` with Zod `.parse` (`real.ts`), mock fallback, and a capability flag (`capabilities.ts`).
- **Dependencies:** no new deps — dockerode (`getArchive`) and the `/v1/shell/exec` path already exist; the durable orchestrator volume already holds `session.log` (`snapshot.ts`).
- **Operational:** retained stopped containers consume disk (~15MB rollout/container) bounded by the 30-day window + free-disk high-water-mark; the corrected reap means Dokploy redeploy / api restart no longer wipes history.
- **Specs:** 2 new (`session-sandbox-retention`, `session-history-replay`) + 3 modified (`aio-sandbox-execution`, `guardrails`, `frontend-console`).
