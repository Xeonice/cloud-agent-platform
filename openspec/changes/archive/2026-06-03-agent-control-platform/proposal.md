## Why

Running coding agents like Codex across multiple tasks today means juggling raw terminals across machines — at 5-10 concurrent sessions this is reported as unmanageable, with no single place to watch what each agent is doing, approve the dangerous things it asks for, or take over its keyboard from a phone. Existing orchestrators (Cloudflare Managed Agents, rivet sandbox-agent, Omnara, Vibe-Kanban) render *structured event streams*, not the real agent TUI, and assume a hosted multi-tenant footprint. We want the opposite: a single-user, self-hostable control plane that drives the **real interactive Codex CLI** and streams its **byte-identical terminal** to a browser, with remote approvals and write-takeover — built backend-first as one greenfield monorepo.

## What Changes

- **Greenfield monorepo from scratch**: a pnpm + Turborepo workspace (`apps/api|web|runner`, `packages/contracts|ui|tsconfig|eslint-config`) with a zod `contracts` package as the single source of truth, strict-TS enforcement in three places (tsconfig, repo Claude Code hooks, husky + lint-staged), and build ordering via `dependsOn: ["^build"]`. Nothing exists yet — no `package.json`, `turbo.json`, configs, or apps.
- **Run the real interactive Codex CLI under a PTY**: the runner spawns the *interactive* `codex` under node-pty and pumps its raw byte stream. exec/app-server/SDK are headless and emit structured JSONL, never the TUI ANSI stream — so byte-identity is a property of the interactive path only, spec'd as **observable conditions** (`TERM=xterm-256color`, matching cols/rows, live-frame parity) and explicitly **NOT** a total promise (scrollback never byte-matches).
- **Byte-identical realtime terminal over WebSocket**: dual-channel WS (raw byte stream + structured control frames) with app-level backpressure — server-side high-water mark (≤ 500K), `requestAnimationFrame` coalescing, `term.write(chunk, callback)` + `pty.pause()/resume()`, and explicit ACK-based pause/resume frames (WS has no native flow control). Reconnect = periodic headless SerializeAddon snapshot (recording cols/rows) + raw append-only `session.log` tail replay.
- **Agent events & remote approvals**: a blocking Codex hook (`PermissionRequest`/`PreToolUse`) forwards the event, blocks, and prints the user's `{decision}` to stdout. The contracts package encodes the `decision.behavior: allow|deny` + optional `message` shape and the **"any deny wins"** rule literally. `PostToolUse` is post-hoc — used for file-edit *reporting* only, with a git-diff fallback because hook tool coverage is partial. Hooks ship via `~/.codex/hooks.json` baked into a runner image pinned to a known-good Codex version; "agent failed to start" is a **distinct task state**, never a hang.
- **Application-layer write-lock with preemptive takeover**: a single-writer/multi-reader lease (`Map<sessionId,{writerClientId,leaseExpiry}>`) with heartbeat and auto-release on disconnect — built in the orchestrator, not delegated to tmux (which has no such primitive). Raw keystrokes require the lock; structured one-shot approvals are **lock-independent**.
- **Notification adapters**: a port with two capabilities — `notify` (one-way push: ntfy/Bark, good for Stop "awaiting input") and `request-decision` (round-trip: Telegram inline buttons via a REST callback, the lock-independent approval path) — since not all channels support the round-trip.
- **Runner dial-back with ephemeral creds**: the runner dials *back* to the orchestrator with a short-lived `TASK_TOKEN` (no inbound port per sandbox); ephemeral sandbox-scoped credentials destroyed with the session are the **primary safety boundary**. The dial-back handshake is a first-class WS frame type in contracts.
- **Three independently-deployable targets**: web on Vercel (web only — Vercel categorically cannot host WebSockets), the stateful NestJS WS+PTY orchestrator on Fly.io or via docker-compose, with `session.log` on a persistent volume that survives orchestrator restart. web and api are independently deployable with env-configurable `API_BASE_URL` / `WS_URL` (never same-origin).
- **Single-user auth, safety guardrails, and a minimal console**: an operator bearer token (distinct from the runner `TASK_TOKEN`) gates all REST + WS access (`/health` open; refuse-to-start when unconfigured); a `MAX_CONCURRENT_TASKS` semaphore plus wall-clock deadline, idle ceiling, and start/turn circuit breaker keep a 7×24 deployment from running away or wedging a slot; and a functional-minimal Next.js console (maintained `packages/ui` with a shadcn-based xterm `<Terminal>`, a session page, a fleet dashboard, and new-task creation) makes the terminal usable. These restore the design doc's `AuthModule` (§4.12) and `BudgetModule`/guardrails (§4.10) and satisfy the shadcn/xterm component-library requirement, while keeping the frontend deliberately thin (no history/settings polish).
- **Deferred behind a port (explicit boundary / non-goal)**: the per-task execution sandbox is a separate, deferred `SandboxProvider` port — the first impl documents that Docker-as-execution forces `--sandbox danger-full-access` (the inner Codex sandbox collapses inside Docker), which is exactly why **Docker is the platform deploy plane, not the per-task execution sandbox**. Also explicitly out of scope: multi-user, token budgeting, and seekable `.cast` history (kept as a deferred-polish page). The frontend stays functional-minimal; this is backend-first.

## Capabilities

### New Capabilities
- `monorepo-foundation`: pnpm + Turborepo greenfield workspace, zod `contracts` package as single source of truth, strict-TS in three enforcement places, repo Claude Code hooks + husky/lint-staged, and a runnable `turbo typecheck lint build` command.
- `repo-and-task-management`: Postgres + Prisma data model and REST API for repos, tasks, and task lifecycle states (including a distinct "agent failed to start" state).
- `terminal-execution`: spawn the real interactive Codex CLI under node-pty in an isolated `workspaces/<id>` with `session.log` as source of truth; "agent failed to start" surfaces distinctly rather than hanging.
- `realtime-terminal`: byte-identical dual-channel WS stream with backpressure (≤ 500K high-water mark, rAF coalescing, ACK pause/resume), and snapshot + tail-replay reconnect under PTY parity conditions.
- `agent-events-and-approvals`: blocking Codex hook forwarder encoding the allow/deny/message contract and "any deny wins" rule, PostToolUse file-edit reporting with git-diff fallback, and the two-capability notification adapter port.
- `write-lock-and-takeover`: application-layer single-writer lease with heartbeat, auto-release on disconnect, and preemptive takeover; keystrokes are lock-gated, approvals are lock-independent.
- `runner-dialback-and-creds`: runner-dials-back handshake authenticated by a short-lived per-task `TASK_TOKEN`, no inbound port per sandbox, ephemeral creds destroyed with the session as the primary boundary.
- `sandbox-provider-port`: a deferred `SandboxProvider` port exposing sandbox-mode as a capability, with a documented minimal Docker impl (`danger-full-access`) and the path to restore OS-level isolation later.
- `multi-target-deploy`: three independently-deployable targets (Vercel web-only, Fly.io / docker-compose api), env-configurable `API_BASE_URL` / `WS_URL` never same-origin, and a required persistent volume for `workspaces/<id>/session.log`.
- `single-user-auth`: an operator bearer token (separate trust domain from the runner `TASK_TOKEN`) gating every REST endpoint (except `/health`) and every client WebSocket connection, with constant-time comparison and refuse-to-start when unconfigured.
- `guardrails`: a concurrency semaphore (`MAX_CONCURRENT_TASKS`) bounding running tasks with FIFO admission, a wall-clock deadline and an idle ceiling that force-fail and reclaim slots, and a circuit breaker on repeated start/turn failure — the 7×24 safety floor (token-level budgeting stays out of scope: pure-terminal precludes it).
- `frontend-console`: a functional-minimal Next.js console — a maintained `packages/ui` (shadcn + Tailwind, incl. an xterm `<Terminal>` wrapper), a session page (live terminal + keystroke input + approval surface), a fleet dashboard, and new-task creation — over env-configurable cross-origin `API_BASE_URL`/`WS_URL`. History/settings polish is deferred.

## Impact

- **New files (greenfield)**: the entire monorepo tree — root `package.json`, `pnpm-workspace.yaml`, `turbo.json`, `tsconfig*.json`, lockfile; `apps/api` (NestJS WS+PTY orchestrator), `apps/web` (Next.js + xterm.js), `apps/runner`; `packages/contracts` (zod schemas + inferred TS, including WS frame protocol and the `ApprovalRequest`/`TASK_TOKEN` handshake), `packages/ui|tsconfig|eslint-config`; `.claude/settings.json` strict-TS hooks; `docker-compose.yml`, `Dockerfile`(s) including the runner image, `fly.toml`, `vercel.json`, and a Prisma schema/migrations.
- **Dependencies**: pnpm + Turborepo, NestJS, node-pty, a pinned known-good Codex CLI baked into the runner image, Next.js + xterm.js (`@xterm/addon-serialize`), zod, Prisma + Postgres (Fly Postgres or compose); optional ntfy/Bark and a Telegram bot for notifications/approvals.
- **External boundaries baked in**: Vercel cannot host WebSockets (web-only); Codex hooks must live in `~/.codex/hooks.json` top-level (repo-local `.codex/config.toml` hook config does not fire); Codex hook coverage is partial (shell + apply_patch + MCP, not all shell calls) so sandbox isolation — not the PreToolUse policy engine — is the primary boundary.
- **Workflow constraints**: this single change is large and will cross `APPLY_PARALLEL_THRESHOLD=12`, triggering track-based parallel-worktree apply — so tasks.md is partitioned into ~9-12 tracks with `monorepo-foundation` as the `depends: none` root, `packages/contracts` edits kept out of parallel feature tracks (routed to a serial integration track) to avoid merge-hell, and `realtime-terminal` / `agent-events-and-approvals` depending on `terminal-execution`. A runnable `turbo typecheck lint build` command is required so apply's post-merge build-verify (bounded `MAX_REPAIR_ROUNDS=3`) and opsx-verify's dynamic checks have ground truth.
- **Constraints / non-goals**: no multi-user, no token budgeting, no seekable `.cast` history (deferred-polish); the concrete OS-isolating sandbox is deferred behind `SandboxProvider`. The frontend is intentionally functional-minimal (backend-first). `research-brief.md` (and a later `verification-report.md`) remain side-car files outside the artifact graph.
