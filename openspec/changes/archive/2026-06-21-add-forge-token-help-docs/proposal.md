## Why

Operators connecting a GitHub / GitLab / Gitee forge credential in the settings card today get only a one-line scope hint ("需 repo 范围的 PAT") and are left to figure out the rest — which page to open, which scopes to tick, and how an agent working in a terminal can mint the same token. This forces every operator to leave the console and hunt through forge docs, and gives agents no terminal-first recipe at all. A dedicated, in-console help page that documents both a human (one-click prefilled web link) and an agent (terminal command) path per forge removes that friction at the exact moment the operator is stuck — on the forge-credentials card.

## What Changes

- Add a new in-console help page at a `_app`-gated route (`/help/forge-tokens`) that renders trusted, app-authored markdown content explaining how to mint a forge access token with the right scopes.
- Author **dual-audience** content for each of GitHub / GitLab / Gitee — a **human version** (one-click prefilled token-creation web link) and an **agent version** (terminal command), with per-forge anchors (`#github`, `#gitlab`, `#gitee`).
- Use real prefilled deep links where they exist: GitHub fine-grained PAT (`?contents=write&pull_requests=write`, classic `?scopes=repo` fallback) with no host prompt; GitLab (`?scopes=api`, plus `read_repository`+`write_repository` least-privilege) with a required instance host; Gitee deep-links to the creation page only (no query-param prefill exists — scope selection described in prose). Document the GitHub-no-host vs GitLab/Gitee-host and the Gitee-no-prefill asymmetries explicitly.
- Document the agent terminal paths: GitHub `gh auth login --scopes repo` then `gh auth token` (noting the gh-OAuth-token-vs-PAT distinction); GitLab/Gitee ask for instance host (default gitlab.com / gitee.com) before constructing the deep link.
- Introduce the **first markdown-rendering path** in the console: add `react-markdown` + `remark-gfm` to `apps/web`, deliberately **without** `rehype-raw` / `rehype-sanitize` (content is trusted and app-authored, so react-markdown's default JSX-escaping is the security guardrail). Load `.md` source via Vite `?raw` build-time string import.
- Wire two contextual link-insertion points in `forge-credentials-card.tsx`: a per-row "如何申请令牌?" link next to each forge hint, and an in-dialog link near the connect `DialogDescription`, both navigating to `/help/forge-tokens` with a `#<kind>` anchor.
- Keep help-page scope copy consistent with the existing card hints ("需 repo 范围", "需 api 范围", "需 projects + pull_requests 范围") to avoid drift.

Non-goals (explicitly out of scope): no global sidebar / mobile-nav entry (the page is reached contextually from the forge card, not a top-level nav slot); no pixel/visual-baseline manifest row (no OpenDesign HTML baseline exists for a markdown content page); no token injection into the sandbox; no `gh`/`glab` preinstall; no changes to the task-delivery flow; not added to the marketing www site.

## Capabilities

### New Capabilities
- `forge-token-help`: An in-console help page that renders app-authored markdown explaining, per forge (GitHub / GitLab / Gitee) and per audience (human web-link version vs agent terminal version), how to mint an access token with the scopes the forge-credentials connect flow requires, including prefilled token-creation deep links and the host-prompt / prefill asymmetries between forges.

### Modified Capabilities
- `frontend-console`: Add a requirement that the forge-token help page is in the route tree behind the `_app` auth gate and is reachable from the forge-credentials card (per-row link and in-dialog link), navigating to the matching per-forge anchor.

## Impact

- **New deps** (`apps/web/package.json`): `react-markdown`, `remark-gfm` (net-new — zero markdown tooling exists anywhere in the monorepo today). No `rehype-raw` / `rehype-sanitize` / `@tailwindcss/typography`.
- **New files** (`apps/web`): a `routes/_app/help/forge-tokens.tsx` route page, a markdown-renderer component (with a token-based `components={{...}}` map onto existing design tokens rather than the typography plugin), and the `.md` content source file(s) imported via `?raw`.
- **Modified file**: `apps/web/src/components/settings/forge-credentials-card.tsx` (two `<Link to="/help/forge-tokens" hash={kind}>` insertion points — self-contained).
- **Auto-generated**: `apps/web/src/routeTree.gen.ts` (gitignored, regenerated by the TanStack generator; CI / worktrees must regenerate it).
- **Typecheck**: `?raw` import is covered by the `vite/client` types already in `apps/web/tsconfig.json`; add a `declare module '*.md?raw'` ambient type only if `turbo typecheck` complains.
- **No backend / API / sandbox changes**; content is static and SSR-safe (no fetch, no window/clock/random).
