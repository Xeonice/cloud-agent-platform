# Research Brief — session-cockpit-redesign

Synthesis of multi-route research (Web design-system / industry literature, Codebase audit, OpenSpec Archive precedent) supporting the session-page "cockpit" redesign. Each finding below is attributed to its research route. The closing "Implications for the proposal" section converts the evidence into concrete authoring decisions, scope boundaries, and risks.

The change is a **pure-frontend, visual-only** re-skin of the existing session detail page (`/tasks/$taskId`): it does NOT change the terminal data pipeline or any backend contract. The intent is to evolve the pixel-merge-era read-only viewer into a "cockpit" — app-shell sidebar + a three-segment header (task-status H1 + tag rail + single 停止 action), an amber write-before-confirm approval banner, and a self-contained dark terminal window (header with ⋯ menu + 全屏, full-width PTY scrollback, statusline footer).

## Web route — design-system contracts & industry literature

### Geist Status Dot validates the "animate-only-when-in-flight, dot+text, never color-alone" rule — but flags a vocabulary correction
- **Source:** https://vercel.com/geist/status-dot
- **Claim:** Geist's Status Dot animates only while in-flight (BUILDING/QUEUED) and goes static at a terminal state; you must NOT add a separate spinner; and "every state ships with a distinct title and label so colorblind users get the same information" (color-alone is forbidden, hence dot+text). Crucially, Geist restricts the Status Dot **component** to the deployment lifecycle only — for any other status (workflow runs, sandboxes, cron jobs) it says "use a Badge with the canonical state vocabulary instead of repurposing the dot."
- **Why it matters:** The proposal's H1 task-status (运行中 / 等待审批脉冲 / 已停止 / 失败 = `data-state`) is a **non-deployment** status, so per Geist's own rule it should carry **Badge-with-canonical-vocabulary** semantics, not a literal Status Dot component. The amber "等待审批 / pending" pulse is the legitimate animated case; 已停止 / 失败 / 运行中 are terminal/static. This is upstream confirmation that the spec's dot+text + animate-only-when-in-flight is faithful, and a flag that the requirement wording should say **"dot+text in Badge form"** rather than "Status Dot component" to stay Geist-faithful.

### Geist Badge backs the "白底 ring tag, 彩底仅警示" tag-rail grammar
- **Source:** https://vercel.com/geist/badge
- **Claim:** "Map meaning to color: green for healthy, red for error, amber for warning, blue for informational or production, gray for neutral." Badges are static labels ("Don't wire onClick onto them; promote to a Button or link if the user can act"), must not stack icons or nest badges, and "Don't add a checkmark icon for success... the variant carries that signal." A `-subtle` variant set exists to "tone any of them down on dense surfaces."
- **Why it matters:** Confirms the spec's tag rail (分支 / Codex / AIO Sandbox / linux-amd64 as neutral white+ring; 写入前确认 as the only amber chip; 守护栏 as neutral) is exactly the prescribed semantic-color mapping — amber is the only warning-bearing chip, everything else neutral gray. It also confirms tags are **non-interactive** (the 停止 button is correctly the only action), and the "icon + text only, never two icons" rule constrains the git-branch+mono branch chip. The `-subtle` variant is the Geist-sanctioned mechanism for the dense statusline/tag surface — worth citing in the delta wording.

### 2026 agent-UX literature names the IA split (conversation vs. activity) and "interface = audit trail"
- **Source:** https://fuselabcreative.com/ui-design-for-ai-agents/
- **Claim:** Combining a conversation/timeline feed with the live activity stream "produces an interface that fails as both a conversation and an activity tracker." The recommended split is conversation-vs-activity, and for regulated/audit contexts "the agent interface is the audit trail" — avoid a second source of truth. Tool-use disclosure should surface "at the decision moment, not in a collapsed log."
- **Why it matters:** External grounding for the three deletions (session-context-strip, terminal-command-input live path, right-side event timeline). The timeline-panel removal maps to "don't run two truth sources / the terminal scrollback is the audit trail." The "tool-use disclosure at the decision moment" pattern is exactly what the amber approval banner does (surface the write command inline, not buried in logs) — so the banner is a **recognized pattern, not an invention**. Use to justify the IA in the why-section.

### Agent-logging community supplies the strongest external citation for deleting the timeline
- **Source:** https://medium.com/@Nexumo_/when-agent-logs-start-lying-ca5f6775a130
- **Claim:** "Confusing a statement about an action with evidence that an action happened" is a core failure mode, and "many stacks accidentally log intent, not commit." The recommended minimum unit is "one trace = one agent run... your single source of truth." A separate piece warns logs can diverge from reality ("When Agent Logs Start Lying").
- **Why it matters:** Directly supports the proposal's stated reason for deleting the right-side event timeline: "terminal scrollback is ground truth, timeline is a divergent second truth source." A reconstructed timeline derived from a raw PTY would be exactly the "log intent not commit" divergence this literature warns about. This is the strongest external citation for that specific deletion — reference it in the design-decision note.

### Competitor approval models confirm gate-on-write is industry-standard (and show the deferred roadmap)
- **Source:** https://docs.devin.ai/cli/reference/permissions
- **Claim:** Devin auto-approves read-only; "writes and shell commands require your explicit approval"; the prompt offers scoped grants ("allow it once, for the session, or permanently for the project") and four modes (Normal / Accept Edits / Bypass / Autonomous). Cursor gates "destructive commands" (rm -rf, DROP TABLE, git push --force) while reads/tests/builds auto-approve, via `session/request_permission`.
- **Why it matters:** Validates that the proposal's amber "写入前确认 (write-before-confirm)" gate and reject/approve banner is industry-standard (gate writes + shell, auto-pass reads). It concretely demonstrates the richer payloads (scoped allow-once/session/project, force-push detection) the proposal correctly **defers** as backend-contract follow-ups — the current `permission_request` carries only `requestId` + `toolName`. Devin's force-push awareness maps to the spec's deferred "非force" meta line. Confirms the honest-degradation scoping is well-judged, not a gap.

### HITL UX best-practices set the bar for banner content and confirm command-only is the minimum-viable ask
- **Source:** https://www.agno.com/blog/how-to-add-human-in-the-loop-controls-to-ai-agents-that-actually-run-in-production
- **Claim:** "When approval is required, the user should immediately understand what they are approving" — "Delete user john@example.com?" beats generic "Tool execution requires approval." Approval must happen "before side effects, not after" (propose-then-commit). Don't "dump the entire chain of thought or every raw log line into the UI"; surface agent + tool + arguments + why. Use modal/queue surfaces in dashboards, message-style in chat. (Mastra adds the "trust spectrum" framing: gate the irreversible/destructive side.)
- **Why it matters:** Sets the bar for banner content: showing the mono command + a clear "Codex 请求执行写入操作" ask satisfies "make the ask clear," and the single black primary (approve) / ghost (reject) matches the recommended low-noise surface. It also frames the deferred diffstat/commits/查看变更 as the "enough context to decide" enrichment — the banner currently shows command-only, which the literature treats as the **minimum-viable ask**, so degrading gracefully here is defensible.

### Readability research defends the deliberate full-width terminal (rejecting prose max-width)
- **Source:** https://www.uxpin.com/studio/blog/optimal-line-length-for-readability/
- **Claim:** Optimal line length for PROSE is ~50–75 chars (60–70 per the British Dyslexia Assoc.), but "longer lines are better for quick scanning... shorter lines for accuracy," and "digital text at 100 characters per line can be read faster than 25-char lines while retaining comprehension." For terminals, 132 columns is "too wide for comfortable reading of straight text but fine for... viewing log files." (Lobsters: "80-col in 2020 isn't reasonable anymore.")
- **Why it matters:** Evidence-based defense of making the terminal full-width and accepting ultrawide whitespace. PTY/codex output is log-scanning content (the "132-col is fine for log files" case), not prose, so the 50–75ch max-width rule that governs the prompt/H1 text does NOT govern the scrollback. The right framing for the design-decision note: **prose regions stay constrained, the scrollback region is intentionally exempt.**

### xterm.js prior art surfaces concrete ⋯-menu / fullscreen gotchas and an a11y copy risk
- **Source:** https://github.com/xtermjs/xterm.js/issues/216
- **Claim:** Auto-scroll best practice: "if a user scrolls up to view past output and the running program adds new output, the viewport should not scroll and interrupt" — pause-on-scroll-up is expected default, and an explicit "pause scrolling" menu item is a manual override. Fullscreen is typically wired via `requestFullscreen` (F11). xterm.js has a known a11y gap: the a11y tree sets `user-select:none`, which "prevents screen readers from selecting text... and prevents users from copying text" (issues 4656/932).
- **Why it matters:** Informs the terminal header ⋯ menu (复制 / 暂停滚动) and 全屏钮 requirements. The "pause autoscroll on scroll-up" behavior is free and expected — the spec should lean on it; the explicit 暂停滚动 item is the manual pin. The known copy/selection a11y bug is a concrete risk to call out for the 复制 menu item — copy may need a custom handler rather than relying on native selection. Fullscreen via `requestFullscreen` is the standard mechanism.

### Industry "terminal-as-the-medium" thesis positions the whole cockpit bet
- **Source:** https://amux.io/blog/best-terminal-ai-coding-agents-2026/
- **Claim:** The most effective agent tools are terminal-based and rely on "representational compatibility... transparency of agent actions within the interaction medium." Warp 2.0 rebranded the terminal as an "Agentic Development Environment," adding the ability to "resume conversations, manage multiple sessions, and track history across devices" — pain points terminal-only solutions cannot solve.
- **Why it matters:** Positions the proposal's whole bet — a web "cockpit" wrapping a raw PTY codex TUI — as on-trend: keep the terminal as the ground-truth medium (transparency within the medium) while wrapping it with web-only affordances (sidebar, status, approval banner) the bare terminal can't provide. This is the high-level "why a cockpit and not just a terminal" justification, and it confirms the decision to NOT add a second composer/input (the codex TUI owns input) respects "representational compatibility" rather than fighting the medium.
