## MET requirements (re-traced end-to-end as satisfied)

### Requirement: ClaudeCodeRuntime credential injection via env token — MET

Re-traced against `agent-runtime` spec.md:61-82; the skeptic's own evidence confirms
a complete, fail-closed implementation, and the refutation ("uses an intermediate
sourced shell file rather than a direct launch env var") does NOT defeat the spec —
the spec requires `CLAUDE_CODE_OAUTH_TOKEN` be SET in the launch environment, and the
sourced `launch-env.sh` (which the launch line `. <file>`s before `claude` starts,
claude-code-runtime.ts:98) sets exactly that var. The seam choice is an unspecified
implementation detail, not a deviation.

- Token injection + fail-closed on empty: `claude-code-runtime.ts:122-149` —
  `material?.oauthToken` check (line 127), `{ok:false, reason:'runtime not configured'}`
  on empty (line 128), base64 round-trip export + `chmod 600` write (lines 133-147).
- `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN`/`apiKeyHelper` unconditionally unset in the
  sourced snippet (claude-code-runtime.ts:136-138) so a stray key cannot shadow the
  OAuth token (spec scenario "Stray API key is neutralized").
- Fail-closed propagation: `agent-runtime.integration.ts:200-209` throws the reason on
  `ok===false`, and the provider try/catch (`aio-sandbox.provider.ts:326-328`) tears down
  the container and rethrows — no unauthenticated claude process starts (spec scenario
  "Missing token fails closed", at PROVISION time; the create-time variant is the
  separate VR-3 gap).
- Auth source reads `CLAUDE_CODE_OAUTH_TOKEN` (`env-claude-auth-source.ts:35-49`);
  `configured()` returns a boolean only, never the token.
- DI bound + exported: `sandbox.module.ts:78-81` (`CLAUDE_AUTH_SOURCE` → `EnvClaudeAuthSource`).
- Status path leaks nothing: `runtimes.service.ts:74-86` calls only `configured()`.

**Minor gap that does NOT block the primary scenario:** the spec's "Missing token
fails closed" scenario is satisfied at PROVISION time but not at ADMISSION time (the
create-time gate is the unbound-token dead letter tracked as VR-3). The credential
INJECTION requirement itself — the subject of this requirement — is fully met.

## Gap findings (stated requirements not satisfied end-to-end)

1. **Runtime selector never reached the working tree** — the create-task dialog +
   readiness gating (frontend-console spec.md:3-10) exist only in
   `.claude/worktrees/wf_ec3b11c4-c66-5/...`; the working-tree dialog has no selector
   and `runtimesQuery`/`getRuntimes`/`mockRuntimes` have zero consumers. → VR-1.
2. **stopOnWrite still over-promises** — active-gate language remains in
   `new-task-dialog.tsx:543-555/564` and `routes/_app/tasks/new.tsx:447-460`; the
   command preview still emits `--confirm-before-write`. Spec requires removal/advisory
   relabel. → VR-2.
3. **Create-time fail-closed gate is a dead letter** — `AGENT_RUNTIME_REGISTRY_TOKEN`
   and `CLAUDE_RUNTIME_READINESS_TOKEN` (`tasks.service.ts:119/138`) are `@Inject`'d but
   never bound by any module; both `@Optional()` deps are always `undefined`, so a
   `claude-code` create with no token is admitted and fails only at provision instead of
   being rejected at admission (spec 4.2). → VR-3.
4. **`.claude.json` provision-time pre-seed is missing** — only comments reference it
   (aio-sandbox.provider.ts:27/310/901); no code writes the global onboarding + per-
   project trust keys (aio-sandbox-execution spec.md:8-9,21-26; agent-runtime spec.md:49-60).
   → VR-4.
5. **Codex not fully extracted behind the port** — the pty client branches
   `runtime.id === 'codex'` to an inline `launchCodex` path (aio-pty-client.ts:357-380)
   with codex-only fields, bypassing `CodexRuntime`; the spec forbids the shared
   scaffolding branching on agent identity outside the port. → VR-5.
6. **`/runtimes` response-shape mismatch** — backend returns `{ runtimes: [...] }`
   (runtimes.service.ts:60) but the consumer expects a bare array
   (`Array.isArray(body) ? body : []`, real.ts:211), so `getRuntimes()` is always `[]`.
   → VR-6.
